Research & Simulation Project • 2024

Path Planning with Kinodynamic Constraints

Autonomous Aerial Robot Navigation in Darkness and Cluttered Spaces. A modular trajectory planning pipeline designed to run under real-time constraints in simulated unknown environments.

Overview

This project focuses on a real-time trajectory planner for drones operating in unknown environments under kinodynamic constraints. The primary aim is to develop an intelligent navigation system for drones that can operate autonomously in environments that are not known beforehand.

The research specifically addresses the challenge of incorporating kinodynamic constraints considering both kinematic limitations and dynamic constraints. The emphasis on real-time computation is crucial because decisions must be made instantaneously during flight, and path adjustments need to occur immediately when new obstacles are detected.

Core Research Question Investigated:
"How can drones effectively navigate and avoid obstacles in real-time while operating in completely unknown environments?"
System Architecture: Gazebo Simulator environment (top) mapping into EDT3D Local Distance Field (bottom-left) and OctoMap Occupancy Grid (bottom-right).

The Framework

Our framework implements a kinodynamically-constrained RRT planner that synthesizes feasible trajectories by incorporating the drone’s dynamic and kinematic constraints during path computation from start to goal states.

While operating in the exploration phase, our system employs the tree-based RRT planner since it provides efficient sampling-based exploration of configuration space, enables real-time path computation, and naturally incorporates both kinodynamic constraints and information gain metrics for unknown environment navigation.

Sub-path Planning & Pruning: Since the drone only perceives obstacles up to a certain range, we cannot plan the complete trajectory in one shot. Instead:

  • We compute sub-paths up to the visible region and append these sequentially.
  • At the initial configuration, the drone plans a candidate path from start to goal.
  • We extract the sub-path up to the visible boundary and prune the remainder.
  • The boundary node becomes the new start position for the next planning iteration.
  • This cycle repeats dynamically until the goal region lies within the visible area.

OMPL Environment

The 2D and 3D environment was designed as a simple workspace where planning tasks relied solely on the OMPL framework. It served as a baseline to evaluate the planner’s ability to generate kinodynamically feasible trajectories without the added complexity of integrating mapping and collision detection libraries.

  • Evaluations were performed on the 2D planner, taking three different step sizes, where the planner was run 5 times for each condition.
  • It was observed that lower step sizes resulted in higher probability of finding the path, where the success rate was defined on number of sub-paths planned for the complete path.
  • If the number of sub-paths crossed 20 and the planner failed to reach the goal region, the result was classified as FAIL, otherwise it was logged as a SUCCESS.

Gazebo & OctoMap Integration

The second environment was more complex, featuring a group of pillars represented within an **OctoMap** framework. These pillars were modeled as obstacles, providing a realistic and cluttered environment for testing.

The **OctoMap** provided a hierarchical representation of the 3D space, while the **EDT3D** library enabled precise local collision detection by computing distances to obstacles within a defined local window around the drone.

Challenges and Results:

  • Accuracy: The planner effectively avoided all obstacles in the environment in its successful run, showcasing its robust collision avoidance capabilities.
  • Scalability: The modular design of the planner enabled it to handle both simpler OMPL-only scenarios and more complex OctoMap-based environments with ease.
  • Flexibility: The planner adapted seamlessly to varied start and goal configurations while maintaining smooth, kinodynamically feasible trajectories.
The planned path for the environment in Simulation.

Performance & Platform

The integration of OctoMap and EDT3D posed additional computational challenges due to the high demands of processing large maps and performing real-time collision detection.

However, our offline setup allowed the planner to utilize the pre-generated map efficiently, demonstrating robust execution. Performance metrics highlight the relation between step boundaries and execution success rates.

Tools and Platforms: The pipeline is built on robust robotics packages including ROS2, OMPL (Open Motion Planning Library), OctoMap 3D voxel grids, and EDT3D distance transforms, implemented entirely in optimized C++ for maximum compute throughput.

Limitations & Future Work

  • Simulation Focused: Since the main focus is the core planning algorithms, the pipeline is evaluated in simulated ROS/Gazebo environments and has not yet been deployed on physical drone hardware.
  • Processing Latency: The sequential mapping-then-planning architecture introduces processing overhead compared to Reinforcement Learning (RL) approaches, which can learn representations and execute actions concurrently.
  • Static Environments: The planning algorithm assumes obstacle layouts are static, making it unsuitable for highly dynamic spaces with moving obstacles or other cooperative agents.
